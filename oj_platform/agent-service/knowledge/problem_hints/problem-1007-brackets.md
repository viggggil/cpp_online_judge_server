# 1007 Brackets 题解

## 问题理解

每个询问给出一个原串子串 `t_i`。我们要把若干子串两两配对，使 `t_a + t_b` 成为合法括号序列，并且每个询问最多使用一次。

对任意一个括号串，可以用栈消去内部已经匹配的部分，剩下的字符决定它还能和什么串拼接。

## 规约思想

扫描一个括号串：

1. 遇到左括号入栈。
2. 遇到右括号时，如果栈顶是对应左括号，则弹出。
3. 否则这个右括号无法被当前串内部匹配，保留下来。

合法拼接 `A + B` 的条件可以理解为：`A` 的未匹配后缀和 `B` 的未匹配前缀能够互相抵消，而且不能出现类型冲突。

对于多种括号，剩余序列不仅要看数量，还要看类型和顺序。因此需要一种稳定表示。可以用实际剩余串作为 key；为了加速，也可以用双哈希表示剩余串及其反向互补串。

## 配对策略

对每个询问子串，计算它的规约结果：

- 如果规约后为空，说明它本身是合法括号序列。两个空 key 可以互相配对。
- 如果规约结果只含一侧未匹配括号，则它只能和对应的反向互补 key 配对。
- 如果规约结果同时含无法由另一串修复的结构，则不能参与有效配对。

统计每种 key 的数量，再与其互补 key 贪心配对。对于 key 与 complement 相同的情况，贡献为 `count / 2`；否则贡献为 `min(count[key], count[complement])`，并注意不要重复统计。

## 区间快速计算

如果 `m` 和 `n` 很大，不能对每个子串重新线性扫描。常见优化方向：

1. 为原串建立括号匹配关系和前缀信息。
2. 对区间规约结果设计可合并摘要。
3. 用线段树、稀疏表或哈希快速得到子串规约后的 key。

MVP 诊断时，重点识别学生是否误把问题当成单一括号计数；多类型括号必须保证类型和嵌套顺序。

## 复杂度

朴素逐个子串规约可能达到 `O(总区间长度)`，在大数据下不可接受。使用可合并摘要或哈希后，目标复杂度通常应接近 `O((n + m) log n)` 或 `O(n + m)`。

## 常见错误

1. 只比较左右括号数量，忽略括号类型。
2. 只比较数量，忽略嵌套顺序。
3. 把本身合法的子串当成可以单独计入答案，但题目要求必须两两配对。
4. 对互补 key 配对时重复统计。
5. 哈希方案没有处理反向互补，导致拼接方向错误。

## 给学生的分级提示

Level 1：合法括号序列不仅要求数量相等，还要求类型和顺序正确。

Level 2：先考虑一个子串内部能消掉哪些匹配括号，剩下的部分决定它需要什么“互补子串”。

Level 3：把每个子串规约成 key，再统计 key 与其反向互补 key 的配对数量。
